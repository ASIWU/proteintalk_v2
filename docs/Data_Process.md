## 输入数据类型

每个任务通常包含两类数据：

1. 样本信息表：描述样本、药物、细胞、批次、标签等元信息。
2. 蛋白质表达量表：描述每个样本对应的蛋白表达向量。

这两类数据最终必须通过统一的 `sample_id` 对齐。

## 期望交付物

### 1. 每个任务的标准化样本信息表

为每个原始任务输出一个统一格式的 `csv`，用于后续直接读入 dataframe。

### 2. 每个任务的标准化表达量数据

为每个任务输出：

1. 一个以 `sample_id` 为 key 的表达量字典
2. 一个与该字典一致的表达量矩阵
3. 一个蛋白顺序列表

其中蛋白顺序必须固定，且统一使用 `UniProt ID`。

### 3. 全局合并后的元信息文件

在所有任务都完成标准化后，生成统一的 `meta json`，至少包含：

1. `uniprotid -> index`
2. `pertid -> index`
3. `pertid -> smiles`
4. `pertid -> target_protein_list`

如有必要，也可以加入每个任务自身的补充元信息，但核心索引体系必须全局一致。

## 统一标准

### 样本信息表统一规则

1. 以双药任务的字段结构作为主参考格式，单药任务向该格式对齐。
2. 对每个字段明确以下内容：
   - 标准列名
   - 数据类型
   - 缺失值填补规则
   - 是否需要统一大小写
   - 是否需要去空格、去分隔符噪音、标准化分号或逗号
3. 单药数据中不存在的第二药物字段，也必须保留统一列，但按规则填默认值。
4. 任何映射关系都不能凭猜测完成，必须逐文件确认。

建议至少统一并明确以下字段：

1. `sample_id`

如果某个任务无法提供其中部分字段，也需要明确记录“缺失原因”和“填补策略”。

### 表达量数据统一规则

1. 表达量数据最终需要与样本信息通过 `sample_id` 一一对应。
2. 输出格式至少包括：
   - `sample_id -> expression vector` 的字典
   - 对应的二维表达矩阵
   - 对应的蛋白顺序列表
3. 蛋白名必须统一映射为 `UniProt ID`。
4. 每个文件都要单独检查“原始列名/蛋白名如何映射到 UniProt ID”。
5. 不能因为不同文件使用了不同命名方式，就跳过映射或直接混用原始蛋白名。

## 必须逐文件完成的工作

对 `./data/rawdata/` 下的每个数据文件，都需要单独完成下面几件事：

1. 确认该文件属于哪一类任务。
2. 确认它是样本信息表、表达量表，还是二者混合。
3. 写清楚原始列到标准列的映射规则。
4. 写清楚该文件独有的特殊处理逻辑。
5. 提取该文件自己的 meta 信息，供后续全局 merge。
6. 输出可复现的 Python 处理代码。

不要先假设所有文件结构一致，也不要先写一个大一统脚本再回头修补。应当先完成文件级别的梳理，再抽象共用逻辑。

## 特殊处理要求

### Extra Data

额外验证集缺少 baseline proteome，因此需要单独补齐 control 信息。

规则如下：

1. 先从单药、双药和额外补充 baseline proteome 中识别 control 样本。
2. `control` 的定义为：
   - `control` 列的值等于 `control`
   - 或 `control == sample_id`
3. 对缺少 baseline 的额外数据，按以下优先级匹配 baseline proteome：
   - `Machine Match`
   - `Type Match`
   - `Batch Match`
   - `Plate Match`
4. 匹配过程需要可追踪，最好输出匹配来源和匹配层级，方便后续审计。

### PTV1

`PTV1` 数据集必须与其他数据隔离处理，不能混入同一套索引空间或样本集合，除非后续任务明确要求合并。
